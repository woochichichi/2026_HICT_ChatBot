## 프로젝트 개요

### 문제

증권사 상담원이 고객 질문에 답하려면 업무 편람을 직접 찾아봐야 한다. 숙련 상담원은 2~5분, 신입은 그 이상 걸린다. 신입 교육도 선배가 붙어서 3~6개월간 OJT를 해야 하는데, 사람이 부족하다.

### 해결

AI가 편람/FAQ를 즉시 검색해서 답변하고(챗봇 모드), AI가 고객 역할을 맡아 신입을 훈련시킨다(훈련 모드).

- **챗봇 모드**: 상담원이 질문 → FAQ/편람 기반 즉시 답변 + 출처 표시
- **훈련 모드**: AI가 고객 역할 → 신입이 답변 → AI가 채점 + 피드백

### 어떻게 동작하나

```
[챗봇 모드]
상담원 질문 → 임베딩 → ChromaDB에서 유사 문서 검색 → LLM이 문서 참고하여 답변 생성 → 출처와 함께 응답

[훈련 모드]
편람 내용 선택 → LLM이 고객 질문 생성 → 신입이 답변 입력 → LLM이 정답과 비교하여 채점·피드백
```

> **Q. 별도로 AI를 학습(파인튜닝)시키나?**
> 아니다. FAQ/편람 텍스트를 벡터로 변환하여 DB에 저장한 뒤, 질문이 오면 유사 문서를 검색해서 LLM에 전달하는 방식(RAG)이다. 데이터가 바뀌면 재인제스트만 하면 된다.
>
> **Q. 두 모드가 데이터를 따로 쓰나?**
> 아니다. 하나의 FAQ/편람 데이터셋을 공유한다. 챗봇 모드는 "검색→답변", 훈련 모드는 "출제→채점" 용도로 같은 데이터를 다르게 활용할 뿐이다.

### 기술 스택

| 항목 | PoC | 실도입 (폐쇄망) |
|------|-----|------------------|
| LLM | GPT-4o API | 미정 — Qwen3-30B-A3B MoE 후보, 3주차 비교 테스트 후 결정 |
| 임베딩 | text-embedding-3-small | 미정 — bge-m3 후보, 차원 불일치(1536→1024) 주의 |
| 벡터DB | ChromaDB (파일 기반) | 동일 |
| 백엔드 | Python FastAPI | 동일 |
| 프론트엔드 | React | 동일 |

> ⚠️ **실도입 모델은 확정이 아닌 후보**입니다. PoC 기간 중 비교 테스트를 통해 결정하며, LLM 전환을 위한 서비스 추상화 인터페이스를 적용합니다. 상세 내용은 [`docs/api-spec.md` 섹션 4](docs/api-spec.md) 참조.
>
> 각 기술을 왜 선택했는지는 하단 [용어집](#용어집)에 정리되어 있다.

### 핵심 설계 원칙

1. **제목+내용 분리 임베딩** — 유사도 가중치는 파라미터화하여 그리드 서치로 최적값 탐색 (기본값 5:5, 3주차 튜닝)
2. **다중 제목 (Multi-subject)** — 원본/유사/요약 제목으로 동의어 문제 해결
3. **LLM 교체 대비 추상화** — `generate()`, `embed()`, `count_tokens()` 인터페이스로 분리
4. **훈련 모드 정답 안전장치** — RAG 추출 정답 + 수동 확정 정답을 병합하여 채점 신뢰성 확보

## 로컬 세팅

```bash
# 1. 클론
git clone <repo-url>
cd 2026_HICT_ChatBot

# 2. 환경변수
cp .env.example .env
# .env 파일에 OPENAI_API_KEY 입력

# 3. 백엔드 (프로젝트 루트에서 실행)
pip install -r requirements.txt
uvicorn backend.main:app --reload
# → http://localhost:8000/docs 에서 Swagger 확인

# 4. 프론트엔드 (2주차~)
cd frontend
npm install
npm start
# → http://localhost:3000
```

### 환경변수 (.env)

```
OPENAI_API_KEY=your-api-key-here
OPENAI_EMBEDDING_MODEL=text-embedding-3-small
OPENAI_CHAT_MODEL=gpt-4o
CHROMA_DB_PATH=./data/chroma_db
```

> **Q. 프론트 없이 백엔드만 돌려도 되나?**
> 된다. 1주차는 CLI + Swagger(`/docs`)로만 테스트한다. 프론트엔드는 2주차부터.

## 역할 분담

<!-- Why: 2인 팀에서 백엔드/프론트 병렬 개발이 가장 빠르다. 
     팀 리드가 RAG·프롬프트(핵심 로직)를, 승구리가 데이터·UI(병렬 가능 영역)를 맡아 
     서로 블로킹 없이 진행하기 위한 구조. -->

| 영역 | 팀 리드 (Woochi) | 승구리 |
|------|-------------------|---------|
| **백엔드** | FastAPI 골격, RAG 파이프라인, 임베딩 서비스, 채점 로직 | - |
| **데이터** | - | FAQ 정제, PDF 파싱 테스트, 인제스트 스크립트 |
| **프론트** | - | React 채팅 UI, 모드 전환, 출처 표시, 채점 결과 |
| **프롬프트** | 챗봇·훈련 모드 시스템 프롬프트 설계 및 튜닝 | - |
| **인프라** | - | 온프레미스 로드맵 문서, 발표 자료 |
| **공통** | 정확도 테스트, 데모 시나리오 | 정확도 테스트, 데모 시나리오 |

> **Q. 왜 이렇게 나눴나?**
> 핵심 로직(RAG, 프롬프트)과 병렬 가능 영역(데이터 정제, UI)을 분리해서 서로 블로킹 없이 동시 진행하기 위함. 프롬프트는 정확도에 직접 영향을 주므로 한 사람이 일관성 있게 관리한다.

### AI 도구 사용 원칙

<!-- Why: AI(Cursor, Claude) 활용 시 코드가 빠르게 나오지만, 
     "돌아가니까 넘어가자" 식으로 넘기면 팀원이 모르는 블랙박스 코드가 쌓인다. 
     2인 팀에서 한 명이 코드를 모르면 유지보수가 불가능해지므로, 
     이해 가능성과 문서화를 강제하는 규칙. -->

1. AI가 생성한 코드는 작성자가 설명할 수 있어야 한다
2. AI에 작업 위임 전 팀원 합의
3. AI 대화에서 나온 설계 결정은 반드시 문서화 (`docs/api-spec.md` 업데이트)
4. **이 리포의 `docs/api-spec.md`를 각자 AI 도구(Claude, Cursor)에 프로젝트 컨텍스트로 등록** — 양쪽 AI가 같은 스펙을 참조하게 함

> **Q. 왜 이런 규칙이 필요한가?**
> AI로 코드가 빠르게 나오면 "돌아가니까 넘어가자" 유혹이 생긴다. 2인 팀에서 한 명이 코드를 모르면 유지보수가 불가능하다. api-spec.md를 양쪽 AI에 등록하는 이유는, 두 AI 도구 간에 맥락이 공유되지 않아 API 형식이 다른 코드가 나오는 사고를 방지하기 위함.

|------|------|------|

### 핵심 개념

| 용어 | 뜻 | 이 프로젝트에서의 역할 |
|------|-----|------------------------|
| **RAG** (Retrieval-Augmented Generation) | LLM이 답변할 때 외부 문서를 검색해서 참고하게 하는 방식. 모델을 재학습시키지 않고도 최신 데이터로 답변 가능. | 편람/FAQ를 검색하여 LLM에 전달 → 할루시네이션 방지, 출처 표시 가능 |
| **임베딩** (Embedding) | 텍스트를 숫자 벡터(예: [0.12, -0.34, ...])로 변환하는 것. 의미가 비슷한 텍스트는 비슷한 벡터가 됨. | "양도세"와 "주식 양도 세금"이 같은 의미임을 벡터 유사도로 판별 |
| **벡터DB** | 임베딩 벡터를 저장하고 유사도 검색을 수행하는 데이터베이스. | 질문 벡터와 가장 유사한 FAQ/편람 벡터를 빠르게 찾아줌 |
| **인제스트** (Ingest) | 원본 데이터(PDF, JSON 등)를 전처리하여 벡터DB에 넣는 과정. | `scripts/ingest_faq.py` 등으로 실행. 데이터 변경 시 재인제스트 |
| **할루시네이션** (Hallucination) | LLM이 문서에 없는 내용을 그럴듯하게 지어내는 현상. | 프롬프트에 "문서에 없으면 모른다고 답변" 규칙을 넣어 방지 |
| **프롬프트** (Prompt) | LLM에게 보내는 지시문. 역할, 규칙, 출력 형식 등을 지정. | `backend/prompts/` 폴더에 모드별 프롬프트 파일로 관리 |
| **PoC** (Proof of Concept) | 개념 검증. 핵심 기능이 동작하는지 소규모로 확인하는 단계. | 4주간 FAQ + 편람 20건으로 핵심 동작 검증 |

| 기술 | 뭔가 | 왜 이걸 선택했나 | 대안과 비교 |

| **GPT-4o** | OpenAI의 LLM. 한국어 성능, JSON 구조화 출력이 우수. | PoC에서 빠르게 결과를 내야 하므로 API 호출만으로 쓸 수 있는 모델 선택. | Claude, Gemini도 가능하나 function calling 생태계가 가장 성숙 |

| **text-embedding-3-small** | OpenAI 임베딩 모델 (1536차원). 한국어 임베딩 품질이 검증됨. | PoC에서 API 호출로 바로 사용 가능. 한국어 증권 용어에 대한 성능이 무난. | bge-m3(오픈소스, 1024차원) — 실도입 후보. 폐쇄망에서 자체 구동 가능 |
| **ChromaDB** | 오픈소스 벡터DB. 파일 기반으로 별도 서버 없이 로컬에서 동작. | 설치 한 줄(`pip install`), 파일 복사만으로 폐쇄망 이식 가능. PoC 규모에 적합. | Milvus(대규모 적합, 설치 복잡), Pinecone(클라우드 전용, 폐쇄망 불가), FAISS(DB가 아닌 라이브러리, 메타데이터 관리 불편) |

| **React** | JavaScript UI 라이브러리. 컴포넌트 기반 개발. | 승구리 기술 스택(3~5년 경험). 채팅 UI 라이브러리 생태계 풍부. | Vue(개발자 경험 부족), Streamlit(빠르지만 커스텀 UI 한계) |

### 설계 기법

| 용어 | 뜻 | 왜 도입했나 |
|------|-----|-------------|
| **제목+내용 분리 임베딩** | 문서의 제목과 본문을 각각 별도로 임베딩하여 두 개 컬렉션에 저장. 검색 시 양쪽 점수를 가중 병합. | 사람인HR 챗봇에서 검증된 기법(정확도 87%). 제목으로 주제를 잡고, 내용으로 세부 매칭하여 검색 품질 향상. |
| **다중 제목 (Multi-subject)** | 하나의 문서에 원본 제목 외 유사 제목(동의어), 요약 제목(LLM 생성)을 추가. | "양도세" vs "주식 양도 세금" 같은 동의어 문제 해결. 사용자가 어떤 표현으로 질문해도 같은 문서를 찾게 함. |
| **그리드 서치** (Grid Search) | 여러 파라미터 조합을 자동으로 시도하여 최적값을 찾는 방법. | 제목:내용 가중치 비율([5:5, 4:6, 3:7])을 자동 탐색. 수동으로 하나씩 바꿔보는 것보다 체계적. |
| **LLM 서비스 추상화** | LLM 호출 코드를 인터페이스로 분리하여, 모델 교체 시 구현체만 바꾸면 되게 설계. | GPT-4o → Qwen3 전환 시 나머지 코드를 건드리지 않기 위함. |
| **수동 정답 병합** | 훈련 모드 채점 시 RAG로 추출한 정답과 사전에 수동 확정한 정답을 함께 사용. | RAG 정확도가 100%가 아니므로(목표 80%), 데모 시나리오의 정답이 틀리는 사고 방지. |
