위키 → 챗봇 데이터 파이프라인 설계

Confluence 업무편람 수집 + 증분(diff) 인제스트 · 2026-06-10 · HICT 챗봇 과제 1
핵심 원칙 — diff의 본질은 "수집 절감"이 아니라 "임베딩 절감"이다. 매일 전체 페이지를 다시 가져와도, 청크 hash 비교가 변경분만 골라내므로 임베딩 API 호출(진짜 비용)은 변경분만큼만 발생한다. 그래서 REST API가 막혀 있어도 증분 인제스트는 성립한다.

1파이프라인 — 단계별로 클릭해 보세요

키보드 ← → 로도 이동 가능

2수집 계층 — 교체 가능한 3방식 (회사 환경에서 되는 것으로)

A. 로컬 HTML 폴더 확실히 됨

브라우저 "다른 이름으로 저장" 또는 위키 내보내기로 받은 HTML을 폴더에 떨굼.

--source dir --path data/raw/wiki_html/

초기 적재 + API 확인 전 0순위. 수동이지만 PDF 출력보다 훨씬 빠르고 구조 보존.

B. 세션 쿠키 크롤링 가능성 높음

브라우저로 보이는 페이지는 쿠키(JSESSIONID)만 복사하면 HTTP GET 가능.

--source crawl --cookie "..."

매일 배치 자동화의 현실적 1순위. 사내망 내부 통신이라 프록시 무관.

C. Confluence REST API 회사에서 확인

/rest/api/content로 본문+버전 조회, CQL로 변경 페이지 목록만 조회 가능.

--source api --token PAT

되면 최선 — 변경 감지가 서버 제공이라 가장 효율적. 안 돼도 A/B로 동일 결과.

세 방식 모두 출력은 동일: RawDocument(source_id, title, html, url) → 이후 파이프라인(파싱→diff→적재)은 한 줄도 안 바뀜. 나중에 --source 옵션만 교체.

3핵심 설계 결정 3가지

결정내용
chunk_id = 내용 hash
{doc_id}_{sha1(제목+내용)[:16]}		 기존 PDF 인제스트는 {doc_id}_{블록순번} — 문단 하나 끼어들면 뒤 순번이 전부 밀려 전체 재임베딩. hash ID면 변경 감지가 "ID 집합 비교"로 끝남: 새 ID만 임베딩, 사라진 ID만 삭제, 같은 ID는 스킵. rag.py 검색 코드는 ID 형식 무관 → 무수정.
문서 단위 1차 스킵
raw HTML hash		 HTML 통째 hash가 메타DB와 같으면 파싱·청킹조차 안 함. 위키 일일 변경률 1~5% 가정 시 배치의 95%+가 이 단계에서 종료 → 임베딩 호출 95%+ 절감.
SQLite 메타 DB
data/meta.db		 README 구조도에 계획만 있던 stores/meta_sqlite.py 실현. 문서 hash·청크 ID 집합·동기화 이력 보관. ChromaDB와 분리해 "무엇이 언제 들어갔나"를 추적 가능하게.

4메타 DB 스키마

-- 문서 단위: 1차 스킵 판단 + 삭제 감지(목록에서 사라진 문서)
documents (source_id PK, doc_id, title, url, raw_hash,
           last_seen_at, last_synced_at, status)

-- 청크 단위: ID 자체가 내용 hash라서 별도 hash 컬럼 불필요
chunks    (chunk_id PK, doc_id, created_at)

-- 배치 리포트: "오늘 뭐가 얼마나 바뀌었나" 추적
sync_runs (run_id PK, started_at, mode, docs_total, docs_changed,
           chunks_added, chunks_deleted, embed_calls)

python scripts/sync_manual.py --source dir --path data/raw/wiki_html/

# 매일 배치 — Windows 작업 스케줄러 등록, 같은 명령 반복 실행

# 크롤링/API 확인되면 --source만 교체 (파이프라인 동일)
