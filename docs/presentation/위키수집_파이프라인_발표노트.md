# 위키 → 챗봇 데이터 파이프라인 — 발표 노트

> 2026-06-12. 편람을 챗봇이 먹게 하는 방법 + 범용화 + 실전 시행착오.
> 상세 설계: [api-spec.md 섹션 9·10](../api-spec.md), [design-ingest-pipeline.html](../design-ingest-pipeline.html), [ADR 0010](../adr/0010-wiki-diff-ingest.md)

---

## 1. 문제 — 편람을 어떻게 챗봇에 넣나

- 업무편람이 사내 위키(Confluence)에 있고, 페이지를 하나씩 PDF로 출력하는 건 비현실적
- 편람은 계속 바뀜 → 매번 전체를 다시 처리하면 비용·시간이 감당 안 됨
- 폐쇄망 + 회사 PC에 Python도 없음

---

## 2. 편람을 "어떻게 저장하나" (핵심)

원문을 그대로 복사해 두는 게 아니라, **검색 가능한 조각으로 변환해서** 저장한다.

```
위키 HTML
  │  ① 파싱   (confluence_html.py) — 제목/문단/표/공지로 블록화 + 계층(가>ㄱ>a)
  ▼
ParsedDocument (공통 규격)
  │  ② 청킹   (ingest.py) — 의미 단위로 묶고 chunk_id = 내용 해시
  ▼
청크 N개
  │  ③ 임베딩 (Gemini) — 청크를 숫자 벡터로
  ▼
저장 3종 (각각 역할이 다름)
  ├─ ChromaDB (벡터DB)   : faq_titles + faq_contents — 의미 기반 검색
  ├─ BM25 키워드 인덱스   : 고유명사·코드(K-OTC, 80~89) 정확 매칭
  └─ SQLite (meta.db)    : 무엇이/언제 들어갔나 + 변경 감지(diff)
```

저장되는 실체:
- **청크 1개** = 제목 벡터 + 내용 벡터 + 메타데이터(출처 문서, 페이지, 계층 경로)
- 질문이 오면 → 질문도 같은 방식으로 벡터화 → 가장 가까운 청크를 찾아 LLM에 전달

> 한 줄 요약: **"편람을 통째 저장"이 아니라 "찾을 수 있는 조각 + 그 조각의 의미좌표(벡터) + 출처"로 저장.**

## 3. 범용적으로 쓰는 법 — 3계층 분리

이 파이프라인의 핵심은 **세 층을 독립**시킨 것. 한 층을 바꿔도 나머지는 그대로.

```
[수집 계층]          [파싱 계층]              [적재 계층]
어디서 가져오나   →   무슨 포맷이냐       →   청킹·임베딩·검색·증분
─────────────       ──────────────         ──────────────────────
로컬 HTML 폴더       Confluence HTML        (포맷·소스 무관, 공통)
쿠키/Windows 크롤    PDF (docling)          - 내용해시 chunk_id
(추가: SharePoint,   (추가: Word, MD,        - diff 증분 갱신
 파일서버, Notion)    일반 HTML)             - 벡터 + BM25 검색
```

**접착제 = `ParsedDocument` 공통 규격.**
- 어떤 소스든 → `RawDocument(html)` 로 통일 (수집 계층의 출력)
- 어떤 포맷이든 → 파서가 `ParsedDocument(blocks)` 로 통일 (파싱 계층의 출력)
- 그 뒤(청킹·diff·임베딩·검색)는 **한 줄도 안 바꾸고 재사용**

**범용화 체크리스트** (새 대상이 생겼을 때):
| 새로 생긴 것 | 추가할 것 | 안 바꿔도 되는 것 |
|--------------|-----------|-------------------|
| 다른 위키/포털/파일서버 | 커넥터 1개 (`connectors/`) | 파싱·적재 전부 |
| 다른 문서 포맷(Word 등) | 파서 1개 (`parsers/`) | 청킹·diff·검색 전부 |
| 다른 임베딩/LLM 모델 | `embedder.py` 구현체 1개 | RAG·검색 로직 전부 |

→ **content-hash chunk_id** 덕분에, 어떤 소스를 붙이든 "변경분만 갱신"이 공짜로 따라온다.

## 4. 실전 시행착오 (발표용 — 실제로 부딪혀 푼 것)

| 문제 | 해결 | 교훈 |
|------|------|------|
| 회사 PC에 Python·pip 없음(폐쇄망) | 수집(PowerShell)↔적재(Python) **머신 분리** | 내부망이 꼭 필요한 최소 단계만 떼면 풀림 |
| 쿠키 수동 복사가 번거로움/오류 | **Windows 통합인증**으로 쿠키 0 통과 | 인트라넷은 쿠키보다 Windows 세션이 되는 경우 많음 |
| 위키 트리가 AJAX라 링크 안 보임 | rootPageId 자동추출 + 트리 API 재귀 순회 | 정적 HTML만 보면 접힌 페이지를 놓침 |
| 정상 페이지를 '로그인 만료'로 오판 | 본문 스캔 제거 → 리다이렉트 URL만 체크 | 정상 페이지에도 로그인 폼 마크업이 있음 |
| 매번 전체 재임베딩은 비현실적 | 내용해시로 **변경분만** 임베딩 | diff의 본질은 수집 절감이 아니라 임베딩 절감 |
| 고유명사·코드 검색 약함(벡터만) | **Hybrid Search**(BM25+벡터 RRF) | 벡터는 의미, 키워드는 정확매칭 — 둘 다 필요 |

## 5. 성과 수치 (PoC, 실데이터)

- 실데이터 편람 페이지 → **401 청크** 적재 (10페이지 샘플), 전체 공간 **약 520페이지** 수집
- 검색 정확도(16문항): **hit@1 62% → 81%**, MRR **0.79 → 0.88** (Hybrid Search 적용)
- hit@5 100% — 정답이 상위 5개 안에는 항상 포함
- 증분: 안 바뀐 문서는 파싱조차 스킵, 일일 변경률 1~5% 가정 시 **임베딩 호출 95%+ 절감**

---

## 6. 한 장 요약 (발표 마무리용)

> 편람을 **"조각 + 벡터 + 출처"로 저장**하고,
> **수집·파싱·적재를 분리**해 어떤 위키·포맷이 와도 재사용되며,
> **내용해시 diff**로 매일 변경분만 갱신하고,
> **벡터+키워드 Hybrid**로 검색 정확도를 끌어올린다.
